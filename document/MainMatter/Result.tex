\chapter{Validación de la aplicación}\label{chapter:result}

En el Capítulo \ref{chapter:implementation} se describieron los detalles de implementación del sistema: la arquitectura, las tecnologías empleadas y las decisiones de diseño. Sin embargo, la implementación por sí sola no garantiza que el sistema cumpla con los requerimientos definidos ni que ofrezca una experiencia adecuada a sus usuarios. Este capítulo somete el producto a un proceso de validación que abarca cuatro dimensiones complementarias: corrección funcional, rendimiento bajo carga, calidad del despliegue web y usabilidad percibida.


\section{Propósito de la Validación}

La validación del sistema persigue demostrar cuatro afirmaciones concretas:

\begin{enumerate}
    \item \textbf{Los flujos principales operan correctamente.} El registro de reportes, la asignación a revisores, la gestión de revisiones y las notificaciones automáticas funcionan conforme a los requerimientos funcionales definidos en el Capítulo \ref{chapter:proposal}.
    \item \textbf{El sistema soporta la concurrencia esperada.} Los endpoints críticos —creación y búsqueda de reportes— responden bajo carga concurrente sin superar los umbrales de tiempo definidos, incluso con volúmenes de datos acumulados.
    \item \textbf{La calidad web cumple con los estándares.} El frontend desplegado satisface los criterios de rendimiento, accesibilidad, mejores prácticas y SEO evaluados mediante Lighthouse.
    \item \textbf{Los usuarios finales consideran la interfaz útil y fácil de usar.} La evaluación mediante el cuestionario SUS (System Usability Scale) debe arrojar una puntuación superior a 68, umbral que define la usabilidad aceptable según Brooke \cite{brooke1996sus}.
\end{enumerate}

\section{Enfoques de Validación}

Para cubrir las cuatro afirmaciones anteriores, se aplicaron los siguientes enfoques complementarios:

\begin{itemize}
    \item \textbf{Pruebas funcionales:} ejecución de los flujos principales del sistema, verificando que cada paso produce el resultado esperado según los requerimientos. Las características del entorno de ejecución se detallan en cada caso de prueba.
    \item \textbf{Pruebas de rendimiento:} evaluación de la capacidad del sistema bajo carga concurrente mediante k6, con el backend y el cliente en máquinas independientes conectadas por red hotspot. Las especificaciones del hardware se presentan en la Sección \ref{sec:entorno-pruebas}.
    \item \textbf{Validación de calidad web:} auditoría del frontend desplegado en Vercel mediante PageSpeed Insights (Lighthouse), evaluando rendimiento, accesibilidad, mejores prácticas y SEO.
    \item \textbf{Pruebas de usabilidad:} recorrido cognitivo con 5 participantes representativos de los cuatro perfiles del sistema, complementado con el cuestionario SUS.
\end{itemize}

\section{Entorno de Pruebas de Rendimiento}
\label{sec:entorno-pruebas}

Las pruebas de rendimiento se ejecutaron con k6, herramienta de código abierto para pruebas de carga. El backend se desplegó en una máquina con sistema operativo Linux Ubuntu 22.04, procesador Intel Core i5 de 4 núcleos a 2.5 GHz, 8 GB de RAM y almacenamiento SSD de 256 GB. k6 se ejecutó en una máquina cliente independiente con Windows 11, procesador Intel Core i7 de 8 núcleos a 3.0 GHz y 16 GB de RAM, conectadas mediante red hotspot a 2.4 GHz. Esta separación garantiza que las mediciones no se vean afectadas por competencia de recursos entre el cliente de pruebas y el servidor.

Se emplearon datos simulados que representan solicitudes realistas de creación y búsqueda de reportes ESAVI, con una carga útil equivalente a la del sistema en producción. Las métricas analizadas fueron: latencia promedio y percentil 95, throughput y tasa de solicitudes exitosas. Se definió como umbral de tiempo aceptable 3000 ms y como criterio de fallo funcional cualquier código de estado distinto de 2XX.

\subsection{Rendimiento}

La categoría de rendimiento mide la velocidad de carga y la capacidad de respuesta de la página. Para ello, Lighthouse emplea las siguientes métricas:

\begin{itemize}
    \item \textbf{Primer Despliegue de Contenido (FCP, por sus siglas en inglés):} tiempo que transcurre hasta que el navegador muestra el primer elemento visual de la página.

    \item \textbf{Despliegue del Contenido Más Grande (LCP):} tiempo necesario para renderizar el elemento de mayor tamaño visible para el usuario, siendo el principal indicador de velocidad de carga percibida.

    \item \textbf{Tiempo Total de Bloqueo (TBT):} tiempo durante el cual el hilo principal del navegador permanece bloqueado e incapaz de responder a las interacciones del usuario.

    \item \textbf{Índice de Velocidad (SI):} mide la rapidez con la que el contenido visible de la página se muestra progresivamente.

    \item \textbf{Desplazamiento Acumulado del Diseño (CLS):} cuantifica la estabilidad visual de la interfaz, reflejando los desplazamientos inesperados de los elementos durante la carga.
\end{itemize}

\end{itemize}

Estas oportunidades de mejora no se corrigieron durante el desarrollo de esta tesis porque no constituyen fallas del sistema, sino optimizaciones de configuración del entorno de despliegue. \subsection{SEO}

Esta categoría analiza qué tan optimizada está la página para los motores de búsqueda. El sistema obtuvo \textbf{83/100} de forma uniforme en todas las páginas y entornos. La auditoría detectó dos carencias puntuales:

\begin{itemize}
    \item \textbf{Ausencia de meta descripción:} el documento HTML no incluye una etiqueta \texttt{<meta name="description">}, utilizada por los buscadores para generar el fragmento descriptivo en los resultados.

    \item \textbf{Archivo robots.txt no válido:} Vercel, al no encontrar un archivo \texttt{robots.txt} en el proyecto, devuelve el \texttt{index.html} en su lugar, lo que genera errores de sintaxis en la auditoría. La solución consistiría en agregar un archivo \texttt{robots.txt} básico que indique las reglas de rastreo.
\end{itemize}

Ambos problemas son superficiales y no afectan la operatividad del sistema. Con estas dos correcciones menores se alcanzaría un puntaje superior a 90.

El sistema cumple satisfactoriamente con los estándares de calidad web en ambos entornos, con puntuaciones de excelencia en escritorio y valores aceptables en móvil. Las incidencias detectadas son superficiales y no comprometen la operatividad ni la experiencia del usuario.


\section{Pruebas de usabilidad}


La usabilidad del sistema se evaluó mediante un recorrido cognitivo con 5 participantes, complementado con el cuestionario SUS (System Usability Scale). Según Nielsen, 5 usuarios son suficientes para identificar aproximadamente el 85\% de los problemas de usabilidad en una evaluación formativa \cite{nielsen2000}. Los participantes se dividieron en dos perfiles: 3 ciudadanos (usuarios del módulo público de creación y búsqueda) y 2 jefes municipales (usuarios del dashboard autenticado).

Cada participante realizó tres tareas guiadas sin recibir instrucciones previas sobre el funcionamiento del sistema. Se registraron las siguientes métricas: completitud de la tarea (éxito/fallo), tiempo de ejecución y dificultades observadas. Al finalizar el recorrido, cada participante respondió el cuestionario SUS, compuesto por 10 afirmaciones con escala de Likert de 1 a 5.

\subsection{Cuestionario SUS}

Las 10 afirmaciones del cuestionario SUS aplicadas fueron:

\begin{enumerate}[label=\textbf{\arabic*}.]
    \item Creo que me gustaría usar este sistema frecuentemente.
    \item Encontré el sistema innecesariamente complejo.
    \item Pensé que el sistema era fácil de usar.
    \item Creo que necesitaría ayuda de un técnico para usar este sistema.
    \item Encontré las funciones del sistema bien integradas.
    \item Pensé que había demasiada inconsistencia en el sistema.
    \item Imagino que la mayoría de las personas aprendería a usar este sistema rápidamente.
    \item Encontré el sistema muy incómodo de usar.
    \item Me sentí muy seguro al usar el sistema.
    \item Necesité aprender muchas cosas antes de poder usar el sistema.
\end{enumerate}

Para las afirmaciones impares (1, 3, 5, 7, 9), el puntaje es la valoración menos 1. 
Para las pares (2, 4, 6, 8, 10), es 5 menos la valoración. 
El resultado se multiplica por 2.5 para obtener una escala de 0 a 100.

\[
\text{SUS} = 2.5 \times \left[ \sum_{i=1,3,5,7,9} (v_i - 1) + \sum_{i=2,4,6,8,10} (5 - v_i) \right]
\]
